OCR 파서 검증 PoC 개발 기획서
PDF·이미지 기반 OCR/문서 파서 자체개발 가능성 검토용
문서 목적
본 문서는 완성형 OCR 솔루션 개발이 아니라, 내부 개발 가능성을 빠르게 검증하기 위한 단일 화면 PoC 개발 방향을 정의한다. Cursor 등 AI 코딩 도구가 화면과 기능 의도를 이해할 수 있도록 기능 범위, 화면 구성, API, 데이터 구조, 개발 순서를 정리한다.

1. 개발 배경 및 목표
1차 개발은 OCR 제품을 완성하는 것이 아니라, PDF와 이미지 문서를 대상으로 내부에서 OCR/파서 처리가 가능한지 빠르게 확인하는 검증 도구를 만드는 것이 목표이다.
웹 화면에서 파일을 선택하고 파일 정보를 확인한다.
선택한 파일 확장자에 따라 사용 가능한 파서 목록을 표시한다.
사용자가 파서를 선택한 후 실행한다.
텍스트, JSON, 로그, 오류, 표 결과를 한 화면에서 확인한다.
여러 파서 결과 비교는 후순위로 두고, 1차에서는 화면 탭과 안내만 제공한다.
개발 원칙
Simple is best. PoC 단계에서는 로그인, DB 저장, 권한, 관리자, 문서 이력 관리, 검수 UI를 제외하고 단일 화면과 단순 API 중심으로 구현한다.

9. 백엔드 개발 방향
PoC 백엔드는 Python FastAPI 기반을 권장한다. OCR/문서 파서 라이브러리 연동이 Python 생태계에서 상대적으로 단순하며, 단일 API로 빠르게 검증하기 좋다.
구성 요소
역할
FastAPI
프론트엔드에서 파일과 파서 ID를 받아 파서 실행
Parser Registry
확장자와 parser_id 기준으로 실행 가능한 파서 관리
Parser Adapter
PDF_TEXT, PDF_IMAGE_OCR, IMAGE_OCR, TABLE_OCR 등의 공통 인터페이스 제공
Temporary Storage
업로드 파일을 임시 저장하고 실행 후 삭제 또는 보관 옵션 적용
Result Formatter
텍스트, 페이지, 표, 로그, 오류를 공통 JSON 구조로 변환
파서 어댑터 개념

아래 프롬프트는 Cursor에 초기 구현을 맡길 때 그대로 사용할 수 있도록 정리한 것이다.
OCR 파서 검증 PoC 웹 도구를 구현해줘.목적:PDF 및 이미지 파일을 웹에서 선택하고, 파일 확장자에 맞는 OCR/문서 파서를 선택 실행하여 텍스트, JSON, 로그, 오류, 표 결과를 확인하는 내부 검토용 PoC이다. 운영 시스템이 아니므로 로그인, DB 저장, 권한, 관리자, 문서 이력, 검수 UI는 제외한다.기술 방향:- Frontend: React + TypeScript + Tailwind- Backend: FastAPI- 화면: 단일 페이지- API: GET /api/health, GET /api/parsers, POST /api/parse필수 기능:1. 파일 선택2. 파일 정보 표시: 파일명, 확장자, 파일 크기, MIME 타입3. 사용 가능 파서 목록 표시4. 파서 선택5. 파서 실행6. 처리 결과 요약 표시: 상태, 사용 파서, 실행 시간, 페이지 수, 추출 문자 수, 표 개수, 오류 개수7. 텍스트 결과 탭8. JSON 결과 탭9. 로그 탭10. 오류 탭11. 표 결과 탭12. 파서 비교 탭은 후순위 안내만 표시파서 ID:- PDF_TEXT- PDF_IMAGE_OCR- IMAGE_OCR- TABLE_OCR- AUTOUI 구조:좌측 360px 설정 패널에는 파일 선택, 파일 정보, 파서 목록, 파서 선택, 실행/초기화 버튼을 배치한다.우측에는 처리 결과 요약 카드와 결과 탭을 배치한다.디자인은 흰색 배경, 연한 회색 카드, 파란색 포인트 컬러를 사용한 깔끔한 개발자용 SaaS 도구 느낌으로 만든다.우선 mock 데이터로 화면을 구현한 뒤, /api/parsers와 /api/parse를 순차적으로 연동한다.
17. 결론
1차 개발 결론
이번 PoC는 OCR 제품 개발이 아니라 파서 구조와 처리 가능성 검증이 핵심이다. 따라서 단일 화면, 단일 파일, 선택 파서 실행, 결과 확인 구조로 최소화한다. 이 PoC 결과를 바탕으로 실제 OCR 솔루션으로 확장할지, 어떤 OCR 엔진과 파서 전략을 채택할지 판단한다.
